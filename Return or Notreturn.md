ถ้าเป็น Arrow Function แบบบรรทัดเดียว (ไม่มีปีกกา) มันมี Implicit Return (ส่งค่ากลับให้อัตโนมัติ) อยู่ในตัวครับ แต่สำหรับงานนี้เรา ไม่ต้องการค่า Return ครับ
ลองดูข้อแตกต่างชัดๆ แบบนี้ครับ:
1. แบบไม่มีปีกกา (Implicit Return)

onClick={() => setAnimal("Dog 🐶")}


* มันแอบ return ไหม? : ใช่ครับ มันจะพยายาม return ผลลัพธ์ของ setAnimal ออกมา (ซึ่งปกติ setState จะ return เป็น undefined)
* มีผลเสียไหม? : ไม่มีครับ เพราะ onClick ของ React มันแค่ต้องการรันฟังก์ชันเฉยๆ มันไม่ได้เอาค่าที่ return ไปทำอะไรต่อ

2. แบบมีปีกกา (Explicit Return)

onClick={() => { 
  setAnimal("Dog 🐶"); 
}}


* มัน return ไหม? : ไม่ return ครับ ถ้าคุณไม่เขียนคำว่า return ไว้ข้างใน ปีกกาจะมองว่าเป็นแค่ "บล็อกของคำสั่ง" (Code Block) ที่ให้รันตามลำดับเฉยๆ

------------------------------
💡 สรุปให้จำง่ายๆ:

* ถ้าเขียน () => setAnimal(...) = รันเสร็จแล้วส่งค่ากลับ (ส่ง undefined กลับไปเล่นๆ)
* ถ้าเขียน () => { setAnimal(...) } = รันเสร็จแล้วจบกัน ไม่ส่งอะไรกลับ

ในเคสของ Event Handler อย่าง onClick เราเน้นให้มัน "ทำงาน" (Action) ไม่ได้เน้น "ค่าตอบกลับ" (Return Value) ดังนั้นจะเขียนแบบไหนก็ได้ตามใจชอบเลยครับ
สงสัยเรื่องการใช้ Return ในฟังก์ชันประเภทอื่นอย่างพวก .map() หรือ .filter() ด้วยไหมครับ? อันนั้นจะสำคัญกว่าในเรื่องการ Return ค่าครับ



ในโลกของ React และ JavaScript สรุปให้จำง่ายๆ แบบนี้ครับ:
1. ไม่ Return (ใช้เพื่อ "สั่งให้ทำ" หรือ Action)
เราไม่ต้องการค่าตอบกลับ แต่ต้องการให้โปรแกรม "เกิดการเปลี่ยนแปลง" บางอย่าง

* เมื่อไหร่: ใน Event Handler เช่น onClick, onChange, onSubmit
* ตัวอย่าง: สั่งให้เปลี่ยน State (setAnimal), สั่งให้แจ้งเตือน (alert), สั่งให้บันทึกข้อมูลลง Database
* ทำไม: เพราะปุ่มกด (Button) ไม่ได้รอเอาค่าอะไรจากฟังก์ชันเราไปใช้ต่อ มันแค่ต้องการให้คำสั่งข้างในทำงานให้เสร็จก็พอ

------------------------------
2. ต้อง Return (ใช้เพื่อ "คายค่าออกมา" หรือ Data Transformation)
เราต้องการผลลัพธ์จากฟังก์ชันนั้น เพื่อเอาไป "แสดงผล" หรือ "คำนวณต่อ"

* เมื่อไหร่:
* ใน .map(): เพื่อบอก React ว่าจะเอาข้อมูลแต่ละตัวไปแปลงเป็น Component หน้าตาแบบไหน (ถ้าไม่ return หน้าเว็บจะว่างเปล่า)
   * ใน Function คำนวณ: เช่น ฟังก์ชันบวกเลข ถ้าไม่ return ค่าผลรวมออกมา เราก็เอาไปโชว์ไม่ได้
   * ใน Component: ตัว App() เองก็ต้อง return (<div>...</div>) เพื่อบอก Browser ว่าจะวาดอะไร
* ตัวอย่าง:

// ต้อง return เพื่อเอา <li> ไปแปะบนหน้าเว็บconst list = items.map((item) => {
  return <li>{item}</li> 
});


------------------------------
💡 วิธีเช็คตัวเอง (Cheat Sheet):

* ถ้าจุดประสงค์คือ "เปลี่ยนค่า/สั่งการ" (เช่น setSomething) $\rightarrow$ ไม่ต้อง Return
* ถ้าจุดประสงค์คือ "อยากได้ผลลัพธ์มาโชว์/ใช้ต่อ" (เช่น map, filter, calculate) $\rightarrow$ ต้อง Return

ลองดูใน Exercise นี้ ถ้าคุณลบ return ใน const App = () => { ... } ออก หน้าจอจะขาวโพลนทันที เพราะ React ไม่รู้จะวาดอะไรครับ
อยากลองดูตัวอย่างการใช้ .map() สร้างปุ่มสัตว์ทั้ง 3 ปุ่มแทนการเขียนแยกทีละปุ่มไหมครับ? (จะได้ฝึกเรื่องการ Return ไปด้วยในตัว)


ค่าที่ใส่ในวงเล็บของ setShow(...) คือ "ค่าใหม่" ที่คุณต้องการให้ตัวแปร isShow เปลี่ยนไปเป็นครับ
คุณสามารถใส่ได้ 3 รูปแบบหลัก ตามการใช้งานดังนี้ครับ:
1. ใส่ค่าที่ต้องการตรงๆ (Direct Value)
ถ้าคุณรู้แน่ๆ ว่ากดปุ่มนี้แล้วอยากให้เป็น true หรือ false ตลอดไป

* setShow(true) → สั่งให้เป็น true ทันที (เปิดไฟ)
* setShow(false) → สั่งให้เป็น false ทันที (ปิดไฟ)
* setShow("Dog 🐶") → (กรณีตัวอย่างแรก) สั่งให้เปลี่ยนเป็นข้อความนี้

2. ใส่ค่าตรงข้าม (Toggle Logic)
ใช้สำหรับปุ่มเดียวที่ทำหน้าที่ทั้งเปิดและปิด (สลับค่าไปมา)

* setShow(!isShow) → อ่านว่า "เอาค่า ไม่ (NOT) isShow ปัจจุบัน"
* ถ้าเดิมเป็น false → !false จะกลายเป็น true
   * ถ้าเดิมเป็น true → !true จะกลายเป็น false

3. ใส่เป็นฟังก์ชัน (Updater Function) - วิธีที่เซฟที่สุด
ในกรณีที่แอปเริ่มซับซ้อน หรือมีการกดปุ่มรัวๆ React แนะนำให้เขียนแบบนี้ครับ:

* setShow(prev => !prev)
* prev คือค่า "ก่อนหน้า" (Previous state)
   * มันจะมั่นใจได้ว่าเราเอาค่าล่าสุดจริงๆ มาสลับ (Toggle) ครับ

------------------------------
⚠️ ข้อควรระวัง (ที่มักจะผิด):

* ❌ ห้ามใส่เครื่องหมายคำพูด setShow("!isShow") → มันจะมองเป็น "ข้อความ" ไม่ใช่ "คำสั่งสลับค่า"
* ❌ ห้ามใส่ปีกกาซ้อน setShow({!isShow}) → ผิดหลักไวยากรณ์ (Syntax Error)
* ❌ ห้ามใส่เครื่องหมายเปรียบเทียบ setShow(isShow == true) → แบบนี้มันจะได้ค่า true/false มาก็จริง แต่มันจะไม่สลับค่าให้เราครับ

สรุปสั้นๆ: ถ้าอยากทำปุ่ม On/Off ให้ใช้ setShow(!isShow) ดีที่สุดครับ!
ลองเอาไปวางในปุ่มแล้วกดดูหรือยังครับว่า ไฟติดๆ ดับๆ หรือยัง?

การเขียนทั้งสองแบบใช้เพื่อทำ Conditional Rendering (แสดงผลตามเงื่อนไข) ใน React ครับ มีความแตกต่างกันดังนี้:
1. แบบ && (Logical AND)
ใช้เมื่อต้องการ "แสดง" หรือ "ไม่แสดงเลย" (มีแค่กรณีเดียวที่สนใจ)

* Logic: ถ้าข้างหน้าเป็น true ให้แสดงตัวหลัง ถ้าเป็น false จะไม่แสดงอะไรเลย
* ตัวอย่าง:

{isOn && <div className="light-bulb"></div>}

แปลว่า: ถ้า isOn เป็นจริง ให้โชว์หลอดไฟ ถ้าไม่จริงก็ไม่ต้องทำอะไร

------------------------------
2. แบบ Ternary (condition ? true : false)
ใช้เมื่อต้องการ "เลือกอย่างใดอย่างหนึ่ง" หรือต้องมีค่า "สำรอง" เสมอ

* Logic: ถ้าข้างหน้าเป็น true ให้ทำหลัง ? แต่ถ้าเป็น false ให้ทำหลัง :
* ตัวอย่าง:

{isOn ? <div className="light-bulb"></div> : null}

แปลว่า: ถ้า isOn เป็นจริง โชว์หลอดไฟ ถ้าไม่จริง ให้ส่งค่า null (ว่างเปล่า)

------------------------------
สรุปความต่าง

| ลักษณะ | แบบ && | แบบ Ternary (? :) |
|---|---|---|
| การใช้งาน | เน้นเช็คเพื่อโชว์/ซ่อน | เน้นการสลับค่า หรือมีค่า Default |
| ความกระชับ | สั้นกว่า (ไม่ต้องใส่ null) | ยาวกว่าเล็กน้อย |
| ตัวอย่างอื่น | {hasError && <p>เกิดข้อผิดพลาด!</p>} | {isLogin ? <LogoutBtn /> : <LoginBtn />} |

อยากให้ช่วยดู Logic ส่วนไหนเพิ่มเติมไหมครับ? หรืออยากลองใช้ Style ในการซ่อนแทนการลบ Element ออกจากหน้าจอ?

``` javascript
import React, { useState } from 'react';

const App = () => {
  const [isOn, setIsOn] = useState(false);

  // 1. ฟังก์ชันที่มีการ Return ค่า (เป็น String)
  const getLightStatusText = () => {
    if (isOn) {
      return "สถานะตอนนี้: เปิดไฟอยู่ (สว่างมาก!)";
    } else {
      return "สถานะตอนนี้: ปิดไฟอยู่ (มืดตึ๊ดตื๋อ)";
    }
  };

  return (
    <div className="container">
      {/* 2. นำฟังก์ชันที่ Return ค่ามาเรียกใช้ใน JSX */}
      <p>{getLightStatusText()}</p>

      <button onClick={() => setIsOn(!isOn)}>
        {isOn ? 'Turn Off' : 'Turn On'}
      </button>

      {isOn && <div className="light-bulb"></div>}
    </div>
  );
};

export default App;
